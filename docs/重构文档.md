# UCAP 企业级化重构文档

**项目名称**: UCAP-agent-demo  
**维护者**: Tom  
**文档定位**: 将教学型 Demo 演进为企业级系统的重构路线与治理标准  

## 1. 背景与定位
企业级不是“能跑”，而是“出了事知道怎么修”。本方案强调稳定性与扩展性是一体两面，先建立可观测与降级体系，再扩展能力边界与数据规模。

## 2. 核心理念
- 契约即代码：契约可验证、可追踪、可回滚
- 数据供应链思维：数据有生产、质检、运输、消费与反馈
- 可观测优先：没有可观测，就没有稳定扩展
- 能力中心：从系统（ERP/HR/FIN）转向能力（Skill）
- 渐进迁移：保留旧路径作为兜底，逐步迁移

## 3. 现状与挑战
- 路由依赖静态映射，契约隐式，扩展需要改 Router
- 错误语义缺失，无法衡量“部分成功”的质量与影响范围
- 配置集中于 .env，不适合多环境治理与敏感信息管理
- 观测缺乏因果追踪，无法定位“意图偏差”与“执行偏差”
- 数据质量未闭环，评分不驱动阻断与反馈

## 4. 企业级化开发计划

### 4.1 第一阶段：可观测基座与语义统一

#### 步骤1：观测语义与链路标准
- [ ] 定义 Trace/Span 命名规范与字段字典
- [ ] 统一 correlation_id 在 API、编排、Agent 之间透传
- [ ] 统一指标语义与错误分类维度

#### 步骤2：三层观测体系
- [ ] APM 层：链路追踪与性能指标
- [ ] LLM 观测层：tokens、工具调用、重试、意图偏差
- [ ] 数据质量层：完整性、一致性、时效性

### 4.2 第二阶段：契约即代码与能力自描述

#### 步骤3：能力契约与自描述
- [ ] 定义 AgentCapability 与 SkillDefinition 能力模型
- [ ] 明确 input/output 的数据契约与版本策略
- [ ] 建立能力注册与发现机制

#### 步骤4：契约测试体系
- [ ] 形成契约基线（Schema 变更触发兼容性检测）
- [ ] 将契约验证作为发布门禁

### 4.3 第三阶段：错误语义与降级体系

#### 步骤5：错误编码与预算体系
- [ ] 定义 error_code/error_class/severity
- [ ] 增加 retryable/fallback_strategy/blast_radius
- [ ] 建立错误预算指标与告警阈值

#### 步骤6：部分成功与隔离策略
- [ ] 明确 degraded 返回语义与缺失标注
- [ ] 建立 Bulkhead 隔离策略与降级路径

### 4.4 第四阶段：Skill 化与路由升级

#### 步骤7：Skill 最小可用单元
- [ ] 将 Agent 作为 Skill 宿主而非业务封装
- [ ] 拆分 1-2 个关键 Skill 作为试点

#### 步骤8：能力路由替代硬编码
- [ ] Router 改为基于能力匹配与权限过滤
- [ ] 保留旧路由作为兜底路径

### 4.5 第五阶段：配置治理与交付策略

#### 步骤9：配置分层
- [ ] 静态配置随代码发布
- [ ] 动态配置可运行时调整
- [ ] 敏感配置集中管理并脱敏审计

#### 步骤10：配置合规与回滚
- [ ] 生产环境禁用 DEBUG
- [ ] 配置变更记录与回滚流程

### 4.6 第六阶段：数据质量闭环

#### 步骤11：质量门禁
- [ ] 质量评分体系与阈值策略
- [ ] 低质量数据阻断与人工审查通道

#### 步骤12：质量反馈
- [ ] 质量评分回写到 Agent 与编排指标
- [ ] 质量问题与源系统关联

### 4.7 第七阶段：事件驱动与读写分离（后续）

#### 步骤13：事件总线与物化视图
- [ ] 事件驱动的数据同步与物化视图维护
- [ ] 质量评分作为事件元数据传播

#### 步骤14：CQRS 与只读视图
- [ ] 写侧负责映射与质量门禁
- [ ] 读侧提供高性能查询与分析
- [ ] CDC 或事件溯源保持一致性

## 5. 高优先级清单（立即执行）
- [ ] 观测语义字典与链路上下文透传
- [ ] LLM 观测指标与意图偏差标注
- [ ] Agent 能力自描述与注册
- [ ] 统一错误编码与部分成功语义

## 6. 验收标准
- [ ] 新增能力无需改 Router 代码即可注册与发现
- [ ] 发生部分失败时可返回可解释结果与缺失标注
- [ ] 观测链路可定位主要故障根因
- [ ] 配置与密钥分层管理满足多环境发布
- [ ] 数据质量问题可阻断且可追溯到来源

## 7. 风险与对策
- 能力拆分过细导致复杂度上升：以“业务动作”为粒度
- 观测体系过重影响性能：先统一语义，再分阶段接入工具
- 迁移影响稳定性：保留旧路径兜底并渐进切换

## 8. 术语表
- Skill：具备输入/输出契约的业务能力单元
- Agent：Skill 宿主与数据源连接管理单元
- 契约：可验证的输入输出规范与版本约束
- 可观测：Trace/Metric/Log/LLM 语义的统一与关联
