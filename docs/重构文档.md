## 开发规范

###  分步骤开发原则

> **⚠️ 分步骤开发原则 ⚠️**
> 
> 1. **严格按步骤执行** - 必须完成当前步骤后才能进行下一步
> 2. **不要重复开发** - 开发前请先详细阅读项目内相关内容，遵循原则：能复用就不要重复开发
> 3. **每步必须测试** - 每完成一个开发步骤，立即进行相应测试
> 4. **测试由Tom执行** - 开发者提供详细测试指导（需要给出适合api测试工具的测试数据以及预期结果等），Tom负责实际测试
> 5. **测试通过后继续** - 只有当前步骤测试通过，才能开始下一步开发
> 6. **问题及时修复** - 如测试发现问题，必须先修复再继续
> 7. **代码质量要求** - 对于代码使用方法不明确是，请先调用Context7阅读相关的官方文档，保证知识是最新的
> 8. **代码注释要求** - 所有代码必须包含必要的注释，包括函数、类、方法等
> 
> **📋 每步开发流程：**
> ```
> 开发实现 → 提供测试指导 → Tom执行测试 → 测试通过 → 下一步
>     ↓                                    ↓
> 如有问题 ← 修复问题 ← 测试失败 ← 反馈问题
> ```

###  测试指导原则

> **🔍 测试指导原则**
> 
> 每个开发步骤完成后，开发者必须提供：
> 1. **详细测试步骤** - 具体的测试操作指导
> 2. **预期结果说明** - 明确的成功标准
> 3. **测试数据准备** - 提供测试用的示例数据
> 4. **问题排查指南** - 常见问题的解决方案


## 1. 背景与定位
企业级不是“能跑”，而是“出了事知道怎么修”。本方案强调稳定性与扩展性是一体两面，先建立可观测与降级体系，再扩展能力边界与数据规模。

## 2. 核心理念
- 契约即代码：契约可验证、可追踪、可回滚
- 数据供应链思维：数据有生产、质检、运输、消费与反馈
- 可观测优先：没有可观测，就没有稳定扩展
- 能力中心：从系统（ERP/HR/FIN）转向能力（Skill）
- 渐进迁移：保留旧路径作为兜底，逐步迁移

## 3. 现状与挑战
- 路由依赖静态映射，契约隐式，扩展需要改 Router
- 错误语义缺失，无法衡量“部分成功”的质量与影响范围
- 配置集中于 .env，不适合多环境治理与敏感信息管理
- 观测缺乏因果追踪，无法定位“意图偏差”与“执行偏差”
- 数据质量未闭环，评分不驱动阻断与反馈

## 10. 链路上下文透传规则（v1）

**生成规则**
- correlation_id 由入口生成，优先使用 request_id
- correlation_id 不可为空，未传入则在入口生成并回填

**透传规则**
- API → 编排 → Agent → LLM 全链路必须透传 correlation_id
- headers 使用统一键名：`x-correlation-id`
- 内部调用上下文统一字段：`correlation_id`

**记录规则**
- 日志与指标必须包含 correlation_id、system、entity_type
- 部分失败必须记录 status=degraded 与 error_class

**关联规则**
- LLM 观测必须记录 tool 调用列表并附带 correlation_id
- Executor 聚合结果必须透传 correlation_id 到返回体

## 11. LLM 专属观测语义（v1）

**记录字段**
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| llm.model | string | 模型名称 |
| llm.tokens | number | token 总量 |
| llm.temperature | number | 采样温度 |
| llm.tools_called | array | 工具调用列表 |
| llm.retry_count | number | 重试次数 |

**intent_execution_mismatch 规则**
- LLM 解析出的 systems 与实际执行 systems 不一致
- LLM 解析出的 entity_type 与返回结果为空且业务可预期非空
- LLM 解析出的时间范围与结果时间分布完全不重叠

## 12. 指标与日志字段统一（v1）

**统一指标名**
| 指标 | 说明 | 标签维度 |
| --- | --- | --- |
| request.duration_ms | 请求总耗时 | system/entity_type/error_class |
| executor.total_duration_ms | 编排总耗时 | system/entity_type/error_class |
| agent.query_duration_ms | 单系统耗时 | system/entity_type/error_class |
| llm.latency_ms | LLM 调用耗时 | system/entity_type/error_class |

**统一降级语义**
- status=degraded：部分系统失败但返回可解释结果
- missing_sources：未返回数据的系统列表
- error_class：按数据源/映射/LLM/网络/超时归类

## 14. 三层观测体系（v1）

**APM 层**
- 采集对象：API/Router/Executor/Agent
- 关键指标：request.duration_ms、executor.total_duration_ms、agent.query_duration_ms
- 关键维度：system、entity_type、error_class、status
- Trace：统一以 query_execution 为主链路入口

**LLM 观测层**
- 采集对象：LLM 调用链路
- 关键指标：llm.latency_ms、llm.tokens、llm.retry_count
- 关键维度：model、tools_called、status
- 语义关联：intent_execution_mismatch 与主链路 trace_id 关联

**数据质量层**
- 采集对象：canonical 映射结果与源系统返回
- 关键指标：quality.pass_rate、quality.missing_rate、quality.staleness_ms
- 关键维度：system、entity_type、status
- 阻断规则：quality.pass_rate 低于阈值触发 status=degraded

## 15. 能力契约与自描述（v1）

**能力模型**
- AgentCapability：system_name、system_type、agent_version、canonical_version、supported_entities、skills
- SkillDefinition：skill_id、name、description、version、input_schema、output_schema、tags

**输入输出契约**
- 输入契约：复用 tools.parameters 的 JSON Schema
- 输出契约：统一输出对象，包含 organizations/persons/customers/transactions 的 JSON Schema
- 版本策略：默认 v1，非兼容变更提升版本号

**注册与发现**
- 注册入口：register_capability(system_key, agent_cls)
- 发现入口：discover_capabilities(systems=None)

## 16. 契约测试体系（v1）

**基线导出**
- export_capability_contracts(systems=None)
- save_contract_baseline(path="data/contract_baseline.json")

**兼容性检测**
- diff_contracts(baseline, current)
- breaking_changes 非空时视为不兼容变更

**发布门禁**
- 若 breaking_count > 0 则阻断发布

## 17. 错误语义与预算体系（v1）

**统一错误语义**
- error_code/error_class/severity/retryable/fallback_strategy/blast_radius
- error_details 与 errors 并存，errors 仅保留原有字符串

**错误预算指标**
- error_budget.total_errors/error_rate
- error_budget.error_class_counts/severity_counts/retryable_count

## 18. 部分成功与隔离策略（v1）

**返回语义**
- status=ok/degraded/error
- partial_success=true 表示部分成功
- missing_sources 记录失败系统

**隔离策略**
- 单系统失败不影响其他系统
- 超时系统标记为 missing_sources
